System overview

The Feynman pipeline converts physics text into a library of Manim-rendered videos suitable for embedding in a physics book. It runs in two parallel tracks that share a backend:

  • Track A — Zone identification: the existing Feynman HTML tool. Runs in the browser. Identifies where in a physics text a visual intervention is needed and what kind.
  • Track B — Video generation: the new pipeline described in this document. Takes zone data from Track A, generates Manim Python, renders it on AWS, and stores the result in a video library with searchable metadata.

The two tracks are connected by a JSON scene spec — a structured object that carries everything Track B needs to generate and render a video without any further human input.

BROWSER Physics text → Feynman tool (existing HTML) → Zone detection output HANDOFF Zone output → "Generate video" action → JSON scene spec (defined below) AWS BACKEND JSON scene spec → Claude API call (generates Manim Python) → ECS/Fargate task (executes Python, renders video) → S3 (stores video + metadata) REVIEW LOOP Render result → Dev checks: does it run? → Physics expert checks: is it accurate? → Patch → re-render → library

AWS architecture — recommendation and rationale

Three options were considered. The recommendation is ECS/Fargate with a containerised Manim environment. The reasoning is below. Developers should flag disagreements in v0.2.

Option Verdict Reason
Lambda Not recommended Manim requires Python, Cairo, ffmpeg, and LaTeX. The combined dependency size exceeds Lambda's layer limits. The 15-minute execution timeout is a real constraint for complex animations. Cold start latency is unsuitable for a review loop that expects fast feedback.
EC2 (persistent instance) Acceptable fallback Simpler to set up. Dev SSHs in to check renders. No container overhead. The tradeoff: you are paying for a running instance even when idle, and horizontal scaling requires manual AMI management. Suitable if render volume is low and the team is small.
ECS / Fargate Recommended Manim + ffmpeg + LaTeX ship cleanly in a Docker image. Fargate tasks spin up on demand and stop when done — no idle cost. Scales horizontally without manual intervention. The official Manim Docker image exists and is maintained. This is the correct environment for a render pipeline that will grow.
Open decision — developer must confirm
ECS/Fargate is the recommendation, but the developer needs to evaluate it against actual AWS account permissions, budget constraints, and whether the team has Docker/ECS experience. This decision gates all subsequent infrastructure work.

Supporting components

API layer

API Gateway + Lambda trigger

A lightweight Lambda function receives the JSON scene spec from the browser tool and queues an ECS task. Lambda is not running Manim — it is only a trigger and message router.

AI generation

Claude API (Anthropic)

Called from inside the ECS task. Receives the JSON scene spec, returns runnable Manim Python. The Claude API key is stored in AWS Secrets Manager, not in source code.

Render environment

ECS / Fargate task

Runs the Manim Docker container. Executes the generated Python. Captures stdout/stderr for the dev review step. Writes the rendered video to S3.

Storage

S3 video library

Stores rendered .mp4 files alongside a JSON metadata file per video. Bucket structure mirrors the library schema (see below). Videos are accessible by the Indiana system via public URLs or signed URLs.

Queue

SQS (optional)

If render volume grows, an SQS queue between the API trigger and ECS prevents task pile-up. Not required for v0.1 testing — add when needed.

Logging

CloudWatch

ECS task logs (including Manim render output and errors) stream to CloudWatch. The dev review step reads these logs to determine if the render succeeded or failed.

JSON scene spec — preliminary schema

This is the data contract between Track A (zone detection) and Track B (video generation). Every field listed here is a proposal — developers and the physics expert should challenge anything that does not serve the pipeline.

The scene spec is generated by the Feynman browser tool when the user requests video generation for a flagged zone. It is passed to the AWS backend as a POST request body.

{
  "scene_id":         "feynman_scene_001",         // unique ID for library lookup
  "version":          "0.1",
  "created_at":       "2026-03-24T10:00:00Z",

  "source": {
    "text_quote":      "As the electron enters the magnetic field...",
    "text_context":    "Chapter 12: Electromagnetism, section 3",
    "book_id":         "physics_101"              // ties video to a specific book
  },

  "zone": {
    "heuristic":       "TD",                       // TD | SG | QR | SC
    "intervention":    "MANIM_ANIMATION",
    "priority":        "Critical"                 // Critical | Important | Supplementary
  },

  "animation": {
    "what_to_show":    "Electron path curving in uniform B field. Show velocity vector, force vector, circular orbit forming.",
    "manim_notes":    "Start with stationary field lines. Introduce electron moving right. Show Lorentz force arrow deflecting path upward. Complete the circular orbit. Label v, F, B throughout.",
    "duration_hint":  30,                         // seconds — hint only, Claude may adjust
    "complexity":     "medium"                   // low | medium | high — affects render time
  },

  "library": {
    "topic":           "electromagnetism",
    "subtopic":        "Lorentz force",
    "keywords":       ["electron", "magnetic field", "Lorentz force", "circular motion"],
    "level":           "undergraduate"            // high_school | undergraduate | graduate
  },

  "status": {
    "render_state":   "pending",                 // pending | rendered | dev_approved | expert_approved | published
    "dev_notes":      "",
    "expert_notes":   "",
    "iteration":      1                          // increments on each patch cycle
  }
}
Schema note
The status block is the review loop in structured form. render_state tracks where in the pipeline a video sits. The dev checks rendered. The physics expert checks dev_approved. expert_approved means the video is ready to publish to the library.

The review loop — how feedback flows

The review loop is a two-gate system. Neither gate is automated in v0.1. Both require human judgment.

Gate 1 — Dev review (technical)

The ECS task logs (CloudWatch) show whether the Manim Python executed without errors and whether a video file was produced in S3. The developer is answering one question: does it run? Not whether it is physically accurate — that is Gate 2.

If the render fails, the developer patches the Python and re-triggers the ECS task. The scene spec's iteration field increments. The patched Python is stored alongside the spec for audit purposes.

Gate 2 — Physics expert review (accuracy)

The physics expert watches the rendered video and answers one question: is this physically correct? Not whether it looks good — whether the motion, vectors, labels, and relationships accurately represent the underlying physics.

Feedback is written into expert_notes in plain language. The developer translates that feedback into Manim changes. The physics expert does not write code.

Workflow gap — needs a UI
In v0.1 both gates are manual — developers read CloudWatch logs, physics experts watch S3 videos directly. This is acceptable for testing on a single scene. It is not acceptable at library scale. A review UI (even a simple HTML page listing pending videos with status controls) is a v0.2 requirement.

Video library — S3 structure and metadata

The video library is an S3 bucket with a predictable folder structure. Each video has a companion .json metadata file in the same folder. The Indiana system will be trained to find and embed videos using the library schema.

s3://feynman-library/
  └── physics_101/
        └── electromagnetism/
              └── lorentz-force/
                    ├── feynman_scene_001.mp4
                    ├── feynman_scene_001.json    // the scene spec, status: published
                    └── feynman_scene_001_v1.py   // the Manim source, version-controlled

The .json file stored with the video is the same scene spec object defined above, updated to reflect final status and any review notes. This means the library is self-documenting — every video carries its own provenance, edit history, and metadata for search.

What this document does not cover

The following are explicitly deferred to v0.2 and later. They are listed here so they are not forgotten:

  • Claude prompt design — the system prompt that instructs Claude to generate Manim Python from a scene spec. This requires testing on real physics scenes before it can be specified.
  • Manim Docker image specification — which base image, which Manim version, what additional LaTeX packages are required.
  • IAM roles and security policy — least-privilege access for ECS task role, S3 bucket policy, Secrets Manager permissions.
  • Error handling — what happens when Claude generates unrunnable Python, when a render exceeds a timeout, when S3 upload fails.
  • Review UI — the HTML interface for dev and expert review workflow.
  • Indiana integration — how the Indiana build system finds and embeds library videos in HTML output.
  • Cost model — ECS/Fargate per-task pricing, S3 storage, Claude API costs per video.

Open decisions — developer response required before v0.2

  1. ECS/Fargate vs EC2: Does the team have Docker/ECS experience? Is there an existing AWS account with ECS enabled? If not, EC2 is the fallback — but confirm now.
  2. Claude API key management: Is the key stored in AWS Secrets Manager, or is there an existing key management pattern in the bearbrown infrastructure?
  3. S3 bucket access: Will rendered videos be publicly accessible URLs or signed URLs? This affects how Indiana embeds them in HTML output.
  4. Review UI: Is a simple HTML review page acceptable for v0.2, or does this need to integrate with an existing workflow tool?
  5. Manim version: Which version of Manim? Community edition (ManimCE) is recommended — confirm the physics expert or a developer has run it locally before we commit to a Docker image.
  6. Scene spec generation: In v0.1 the scene spec is assembled by the Feynman browser tool. Should the developer also be able to write scene specs manually, bypassing the browser tool? This affects schema strictness.

What v0.2 will contain

Once the open decisions above are resolved, v0.2 will be a set of three operational documents:

  1. Deployment guide (how-to) — step-by-step setup of the AWS environment, Docker image build, ECS task definition, S3 bucket policy.
  2. Scene spec reference (reference) — final locked schema with field definitions, allowed values, and examples for each heuristic type.
  3. Review loop protocol (how-to) — instructions for the developer and physics expert on how to move a video from pending to published.

v0.2 docs are written after the first test render succeeds on a single physics scene. Do not write deployment docs for a system that has not been tested.

This document is intentionally incomplete. It is a structured starting point, not a finished specification. Every section marked as an open decision is a blocker for v0.2. The goal of v0.1 is agreement, not completeness.
