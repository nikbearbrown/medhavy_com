Tool overview and file structure

The Feynman tool is a single self-contained HTML file. It has no build step, no dependencies, no server component. It runs entirely in the browser and makes one outbound call: a POST to the Anthropic API.

The file has four functional parts:

PartLocation in fileWhat it does
UI <body> HTML Textarea for input, Analyze button, output div, spinner
SYSTEM const Top of <script> The full system prompt sent to Claude on every request — defines the four heuristics, four intervention types, and the exact output format
analyze() function <script> Reads the textarea, calls the API, writes the response to #output
Output rendering #output div Plain text rendering via white-space: pre-wrap — no Markdown parsing, no HTML injection
API key exposure — known issue
In the current implementation the Anthropic API key is sent from the browser directly to api.anthropic.com. This means the key is visible in browser devtools network tab. This is acceptable for internal use. It is not acceptable for any public-facing deployment. For v0.2 the API call must be proxied through the AWS backend — the browser tool sends the request to an API Gateway endpoint, not to Anthropic directly.

The API call

The analyze() function makes a single POST request to the Anthropic messages endpoint. The full call structure:

const res = await fetch("https://api.anthropic.com/v1/messages", {
  method: "POST",
  headers: {
    "Content-Type": "application/json"
    // Note: no x-api-key header in current implementation
    // API key handling is managed by the browser environment
    // See API key exposure note above
  },
  body: JSON.stringify({
    model:      "claude-sonnet-4-20250514",
    max_tokens: 1000,
    system:     SYSTEM,            // the full system prompt
    messages: [{
      role:    "user",
      content: text                // raw physics text from textarea
    }]
  })
});

Response handling

const data = await res.json();
output.textContent = data.content.map(b => b.text || "").join("\n");

The response is concatenated plain text. No JSON parsing. No Markdown rendering. The system prompt instructs Claude to return plain text only — no asterisks, no pound signs, no bullet dashes. If formatted output appears in the browser, the system prompt format instruction has been overridden or ignored.

ParameterCurrent valueNotes
model claude-sonnet-4-20250514 Sonnet is the correct tier for this task — sufficient reasoning, acceptable latency. Opus is not needed. Haiku will degrade output quality.
max_tokens 1000 Sufficient for short passages. For long texts with many zones this will truncate output. Raise to 2000–4000 if truncation is observed.
system Full SYSTEM const Sent on every request. Changes to the system prompt take effect immediately on next Analyze click — no rebuild required.

System prompt architecture

The SYSTEM const is the entire intelligence of the tool. It defines the persona, the four heuristics, the four intervention types, and the exact output format. It has five sections:

SectionPurposeSafe to modify?
Persona declaration Establishes "You are Feynman, a Physics Visual Architect" — sets the analytical register Yes — change the name or framing without affecting output structure
Four intervention types Defines STATIC DIAGRAM, MANIM ANIMATION, INTERACTIVE SIMULATION, GRAPH OR CHART Yes — add, remove, or rename types here. Any new type must also be added to the output format spec.
Four heuristics Defines TD, SG, QR, SC — the triggers for each intervention type Yes, with caution — changing heuristic definitions changes what gets flagged. Test on known passages before deploying.
Format instruction "Output your analysis in this exact plain-text format. No markdown…" Do not change the no-markdown instruction. The output rendering relies on plain text.
Output template Defines SUMMARY, ZONE DETECTION TABLE, FIGURE DETAILS, DENSITY RECOMMENDATION, PRIORITY RANKING sections with exact field names Modify only if the pipeline consumer is updated to match. The pipeline parses this output — changes here break the handoff.
The fragile part
The output template section is the contract between this tool and the pipeline. The pipe-delimited zone detection table and the Zone [number] / Quote: / Heuristic: field structure are what the pipeline parses to generate scene specs. Change those field names only in coordination with the backend developer.

Output format specification

The tool produces five named sections in order. Each section has a fixed structure. This is the reference for anyone parsing the output programmatically.

SUMMARY

Two to three sentences of plain prose. No fixed subfields. Not machine-parseable — for human review only.

ZONE DETECTION TABLE

One line per zone, pipe-delimited. Field order:

Number | Text Quote (first 8 words) | Heuristic | Recommended Intervention | Rationale

Example:

1 | As the electron enters the magnetic | TD | MANIM ANIMATION | Time-evolving circular orbit cannot be conveyed in text

FIGURE DETAILS

One block per zone. Fixed field sequence:

Zone [number]
Quote: [first 8 words of flagged passage]
Heuristic: [TD / SG / QR / SC]
Intervention: [type]
What to show: [description]
Manim notes: [animation sequence — only present if MANIM ANIMATION]
Simulation parameters: [only present if INTERACTIVE SIMULATION]
Verification checks: [one sentence per check, no bullets]
Conditional fields
Manim notes and Simulation parameters are only present when their intervention type applies. A parser must handle their absence without error.

DENSITY RECOMMENDATION

One to two sentences. States total figure count and mix of types. Plain prose.

PRIORITY RANKING

One line per zone. Format:

[Zone number] — [Critical / Important / Supplementary] — [one-sentence reason]

How to modify the tool

Adding a new intervention type

  1. Add the type name and definition to the four intervention types section of SYSTEM
  2. Add a new heuristic that triggers it, or amend an existing heuristic to include it
  3. Add the new type to the output template's Intervention: field allowed values
  4. Notify the backend developer — the pipeline scene spec will need a new handling branch for the new type

Changing the output format

  1. Edit the output template section of SYSTEM
  2. Test on three known passages and verify the new format is consistently produced
  3. Update the pipeline parser before deploying — the two must stay in sync
  4. Increment the tool version number in the HTML <title> and version block

Adjusting max_tokens

If output is being truncated mid-section, increase max_tokens in the analyze() call. A passage with six flagged zones will produce approximately 1200–1600 tokens of output. Set max_tokens to at least 1.5× the expected output length.

Pipeline integration points

The tool currently has no connection to the Feynman pipeline backend. It is a standalone analyzer. The integration points for v0.2 are:

Integration pointWhat changesv0.2 requirement
API call destination Currently calls Anthropic directly. Must be rerouted to AWS API Gateway endpoint. Required — removes direct API key exposure
"Generate video" action A button per flagged zone that sends the zone's FIGURE DETAILS to the backend as a JSON scene spec. Required — this is the handoff from Track A to Track B
Scene spec assembly The tool must parse the FIGURE DETAILS output and assemble a valid JSON scene spec (as defined in the pipeline architecture doc) before POSTing to the backend. Required — the backend expects structured JSON, not raw text
Status feedback A way for the tool to show whether a video for a given zone is pending, rendered, or published — pulling from the scene spec status.render_state field. Nice to have in v0.2 — not a blocker
Related doc
The JSON scene spec schema and the full pipeline architecture are documented in feynman-pipeline-v01.html. Read that document before implementing the integration points above.

Debugging common failures

SymptomLikely causeFix
Output contains asterisks, pound signs, or bullet dashes Claude is ignoring or overriding the plain-text format instruction Strengthen the format instruction: add "If you use any markdown formatting characters your output is invalid" to the SYSTEM prompt
Output truncates mid-section max_tokens limit reached Increase max_tokens to 2000 or 4000. Check CloudWatch token usage if routing through backend.
No output — spinner does not stop API call failing silently Open browser devtools → Network tab → find the failed POST → read the response body for the Anthropic error message
FIGURE DETAILS fields missing for a zone Claude omitting conditional fields (Manim notes, Simulation parameters) even when applicable Make the conditional fields unconditional in the output template — require them always, allow empty values
Zone detection table has inconsistent pipe count Claude including pipes inside a field value (e.g. a rationale containing a pipe character) Change the delimiter to || or use a numbered field format instead of pipe-delimited
Priority ranking missing zones max_tokens cut the output before the final section Increase max_tokens. Alternatively, instruct Claude to output PRIORITY RANKING before FIGURE DETAILS — this prevents truncation from cutting the most actionable section.

This is a v0.1 reference. The tool is a single-file implementation. Complexity budget is deliberately low — resist the urge to add a build step, a framework, or a backend dependency before the pipeline integration in v0.2 is confirmed.
